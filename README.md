# internal-tool-frontend

## 👨‍🔧 프로젝트 개요

이 저장소는 시설관리 도메인(설비, 소방, 에너지 관리)을 기반으로 한
내부 업무용 웹 도구를 직접 설계·구현하는 학습/실전 프로젝트입니다.

단순 튜토리얼 코드가 아니라, 실제 시설관리 현장에서 사용하는
점검/대시보드 흐름을 반영하는 것을 목표로 합니다.

## 📚 진행 방식 (BLOCK 단위)

이 프로젝트는 **"내부툴 개발자" 커리큘럼**에 따라
BLOCK 단위(예: 004, 005, …, 012…)로 꾸준히 확장하는 방식으로 진행 중입니다.

- 각 BLOCK은 2~3일 분량의 학습·구현 단위입니다.
- BLOCK마다 아래 순서를 반복합니다.
  1. 목표 기능 정의 (예: 로그인 검증, 점검 등록, 목록 렌더링)
  2. HTML/CSS/JS 설계
  3. 구현 및 디버깅 (브라우저/콘솔 확인)
  4. 불필요 코드 정리
  5. `git commit` → `git push`

> 같은 파일을 계속 수정하지만,  
> **Git 커밋 로그와 코드 주석으로 BLOCK 경계가 남도록 관리**하고 있습니다.

## ✅ BLOCK 진행 현황 & 로드맵

### 1단계: 기본 구조 & 인증 (001–009)

- 001–004: 기본 레이아웃/네비게이션, 점검 등록 폼 HTML/CSS, 기초 입력값 추출
- 005–007: 로그인 페이지(auth.html), AND/OR 검증, 권한/작업 조건 가드 패턴 도입
- 008–009: `canUserStartWork` 순수 함수 도입, 사용자 상태 스냅샷(userState) 개념 정리

### 2단계: 점검 데이터 저장 & 목록/상세 (010–014)

- 010–012: 점검 등록 폼 유효성 검사, `localStorage` 저장, 점검 목록 렌더링, 선택 상태(.active), 상세 패널 기본 구현
- 013: 점검 상세 패널 구조 정리 및 상태·비고 표시
- 014: 대시보드 통계 카드(총 건수, 상태별 집계) 구현 및 `getChecksWithFallback` 재사용

### 3단계: 검색/필터/정렬/페이지네이션 UX (015–020)

- 015: 설비명 검색 + 상태 필터(전체/정상/이상/점검중), 필터 상태 관리 변수 도입
- 016: 정렬 옵션(날짜 최신/오래된 순, 설비명 가나다순)과 정렬 로직(`Array.sort`) 추가
- 017: 필터/정렬 결과 개수 표시(`checkCount`) 및 예시 데이터/실제 데이터 구분 문구
- 018: 상태 뱃지(`.status-badge`)와 상태별 색상(`.status-normal / .status-error / .status-check`) 도입
- 019: 점검 목록 페이지네이션(이전/다음, `PAGE_SIZE`, `currentPage`) 및 필터/검색/정렬과의 연동 리팩토링
- 020: 점검 목록 상단 필터 영역 UX 개선(`filter-bar`, `checkCount` 위치 조정, 초기화 버튼 배치)

### 4단계: 점검 수정/삭제 & 상세 UX 강화 (021–025)

- 021: 선택한 점검을 수정할 수 있는 폼과 “수정 모드” 흐름 추가
- 022: 점검 삭제 기능과 삭제 후 목록·상세·대시보드 동기화
- 023: 저장/수정/삭제 과정의 에러 처리 및 사용자 메시지 정리
- 024: 점검 상세 패널 레이아웃과 긴 메모 가독성 개선
- 025: 등록·수정·삭제 이후 공통 렌더링 함수로 리팩토링

### 5단계: 반응형 레이아웃 & 접근성·사용성 개선 (026–030)

- 026: 모바일·태블릿 대응을 위한 기본 레이아웃 브레이크포인트 정의
- 027: 키보드 포커스, 탭 순서, aria 등 기본 접근성 요소 점검
- 028: 로딩/빈 상태/에러 상태를 공통 컴포넌트 스타일로 통일
- 029: 폼 유효성 검사 메시지 구조·스타일 정리로 한눈에 보이게 개선
- 030: 필터/정렬/페이지 계산용 순수 함수에 간단한 테스트 구조 도입

### 6단계: 모듈 분리 & 간단한 API 레이어 도입 (031–035)

- 031: `script.js`를 기능별 파일로 분리하고 각 파일 역할 문서화
- 032: `localStorage` 접근을 감싸는 repository/service 레이어 추가
- 033: 에러 핸들링 로직을 공통 모듈로 묶어 재사용성 향상
- 034: 상수·설정값을 별도 모듈로 분리하고 환경별 설정 구조 설계
- 035: 폴더·파일 구조 재점검 및 불필요 코드·주석·네이밍 정리

### 7단계: 포트폴리오 마감 & 문서화 (036–040)

- 036: README 최종 정리 및 주요 화면 스크린샷·사용 예시 추가
- 037: 시연용 스크립트와 실제 사용 시나리오를 문서로 정리
- 038: 프로젝트 회고(배운 점, 어려웠던 점, 다음 버전 개선점) 작성
- 039: Git 커밋 규칙 `(블럭번호)type: 설명`과 브랜치 전략 회고
- 040: GitHub Pages 등 배포와 포트폴리오용 소개 문구·링크 정리

## Backlog (후반 적용)

### Sidecar 모듈 A: 로딩/실패/재시도 (보류)

- 목적: 비동기/에러처리/UX 증거를 프로젝트에 추가
- 최소요구: loading 상태 + 실패 메시지 UI + 재시도 버튼 1개
- 실패 시나리오(개발모드): 30% 실패 또는 600~1200ms 지연 중 1개를 "한 곳"에만 적용
- 적용 시점: 022 이후 블럭 완료 후, 난이도/진도 안정화 시점에 1블럭으로 묶어서 적용
- (Backlog) 사이드카 A(로딩/실패/재시도), B(Audit Log)는 022~023 흐름 완주 후 후반 리팩토링 블럭에서 적용 예정

(산출물 3줄)

1. 로딩/실패/재시도 UI를 추가해 비동기 실패 시나리오를 처리했습니다.
2. 에러 발생 시 사용자에게 메시지를 제공하고 동일 요청을 재시도할 수 있게 했습니다.
3. 상태값(isLoading/errorMsg) 기반으로 화면이 일관되게 렌더링되도록 구성했습니다.

## ✅ 현재 진행 상황 (BLOCK 기준)

- 001: 점검 등록 폼 HTML 구조
- 목적: 내부툴 프론트엔드 기초부터 누적 기록

- BLOCK 001–004
  - 기본 레이아웃, 폼 구조, 입력값 추출, 기본 검증

- BLOCK 005–007
  - 로그인 페이지(auth.html)
  - AND/OR 조건 검증
  - 권한/작업 조건 가드 패턴 기초

- BLOCK 008–009
  - 작업 가능 여부를 판단하는 `canUserStartWork` 순수 함수 도입
  - 사용자 상태 스냅샷(`userState`) 개념 정리

- BLOCK 010–012
  - 점검 등록 폼(index.html) 유효성 검사
  - 점검 데이터를 `localStorage`에 저장
  - 점검 목록(checks.html) 렌더링 및 선택 상태(`.active`) 구현
  - 선택된 점검 상세 패널, 대시보드 준비

- BLOCK 013
  - 점검 상세 패널 정리
  - 대시보드 통계 카드(총 건수, 상태별 집계) 연결
- BLOCK 014 : 대시보드 요약 카드
  - `localStorage`에 저장된 점검 데이터를 불러와 총 건수/정상/이상 건수 집계
  - 메인 대시보드에서 한눈에 설비 상태를 확인할 수 있는 통계 카드 UI 구성

- BLOCK 015 : 점검 목록 검색/필터
  - 설비명 텍스트 검색 + 상태별 필터(전체/정상/이상/점검중) 기능 추가
  - `getFilteredChecks()`로 검색/필터 로직을 한 곳에서 관리하도록 구조화

- BLOCK 016 : 점검 목록 정렬 옵션
  - 날짜 최신순/오래된순, 설비명 가나다순 정렬 옵션 추가
  - `sortOrder` 상태를 기준으로 `Array.sort()`로 정렬 처리

- BLOCK 017 : 조건에 맞는 점검 개수 표시
  - 현재 검색/필터/정렬 조건에 맞는 점검 건수를 상단에 표시
  - 사용자 입장에서 “지금 보고 있는 목록이 몇 건인지” 직관적으로 확인 가능

- BLOCK 018 : 상태 배지 스타일링
  - 상태 텍스트를 `정상/이상/점검중`에 따라 색상이 다른 배지 형태로 표시
  - `getStatusClass()` 함수로 상태 텍스트 → CSS 클래스 매핑 분리

- BLOCK 019 : 점검 목록 페이지네이션 도입
  - `currentPage`, `PAGE_SIZE`를 도입해 한 페이지당 N개씩 잘라 보여주도록 리팩토링
  - 검색/필터/정렬 결과에 따라 페이지 수 자동 계산 및 이전/다음 버튼으로 이동 가능

- BLOCK 020 : 점검 데이터 초기화 & 필터 영역 UX 개선
  - “점검 데이터 초기화” 버튼으로 예시 데이터/사용자 데이터 재설정 기능 추가
  - 현재 조건에 맞는 점검 개수 표시 + 필터/정렬/초기화 버튼 레이아웃 정리

- BLOCK 021 : 점검 상세 수정/삭제 (Edit Mode)
  - 점검 상세 패널에 수정 모드(editMode) 추가
  - 예시 데이터(SAMPLE_CHECKS)는 수정/삭제 불가 처리
  - 저장 데이터(localStorage)만 수정/삭제 가능

- BLOCK 022 : UX 보정 (선택/페이지/필터)
  - 필터/정렬 결과에서 선택된 항목이 사라지면 자동 선택 해제 + 편집 종료
  - 선택된 항목이 남아 있으면 해당 항목이 포함된 페이지로 currentPage 자동 이동
  - 페이지 이동 시 현재 페이지에 선택 항목이 없으면 선택 해제 처리(UX 안정화)

- BLOCK 023 : 비동기 삭제 시뮬레이션 + 로딩/잠금 + 상태 메시지
  - 삭제 작업을 async/await로 처리하고 서버 요청처럼 랜덤 지연(600~1200ms) 적용
  - isBusy로 중복 클릭 방지(삭제 버튼 잠금/해제)
  - asyncStatus 영역에 “삭제 중/완료/실패” 상태 메시지 표시 + 일정 시간 후 자동 해제

- BLOCK 024: 수정 저장도 비동기 처리(로딩/잠금/상태메시지 자동 해제) 적용
